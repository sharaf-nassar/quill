# Implementation Plan: Active-Time Runtime Tracking Redesign

**Branch**: `008-runtime-redesign` | **Date**: 2026-05-20 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/008-runtime-redesign/spec.md`

## Summary

Replace the user→assistant pair-based `response_times` ingestion with a
per-event session timeline. Every non-meta line of every CC and Codex JSONL
transcript becomes one row in a new `session_events` table, classified by
event kind. The LLM Runtime card is recomputed from this table as a sum of
per-chain active intervals, where idle gaps split logical turns and
tool-loop gaps do not. The existing `response_times` table stays in place
unchanged so the Sessions breakdown and sub-agent tree continue to work
without regression. Backfill happens automatically through the existing
mtime-based reindex sweep via a `runtime_event_reingest_pending` flag in
`settings`, matching the pattern used by migrations 20-22.

## Technical Context

**Language/Version**: Rust 2024 edition (backend), TypeScript 5.x (frontend)
**Primary Dependencies**: Tauri 2, `rusqlite` (WAL mode, 5-second busy
timeout), `chrono`, Tantivy (search index — untouched by this feature),
React 18, Recharts (frontend — sparkline visualization only)
**Storage**: SQLite at `~/.local/share/com.quilltoolkit.app/usage.db`
(adds one table at migration 26)
**Testing**: `cargo test` for Rust (storage + sessions modules carry their
own `#[cfg(test)]` blocks today); manual UI verification via `npm run tauri
dev` for the analytics card
**Target Platform**: Linux, macOS, Windows desktop (Tauri 2)
**Project Type**: desktop-app (Tauri 2: Rust backend + TS/React frontend)
**Performance Goals**: Re-ingesting an entire transcript history of a heavy
developer machine (hundreds of sessions, tens of thousands of events) must
complete inside the existing mtime sweep without blocking the UI thread.
Steady-state runtime query for the 1h window returns in under 100 ms on a
typical machine.
**Constraints**: Migration must be safely re-runnable. The new table must
share `delete_*_for_session` semantics with `tool_actions` and
`response_times` so session-deletion stays atomic. No new background tasks
or threads. No changes to the search index schema. No changes to the
Tauri IPC contract surface for `get_llm_runtime_stats` (existing
`{range, scope}` request and `LlmRuntimeStats` response shape stay).
**Scale/Scope**: Existing developer machines hold roughly 100-2000 CC and
Codex transcripts. Heavy sessions emit on the order of 1000 events. The
table is therefore expected to land in the low millions of rows steady
state — well within SQLite's comfortable range with the proposed indices.
