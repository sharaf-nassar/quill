# Implementation Plan: Hooks Breakdown Tab

**Branch**: `009-hooks-breakdown-tab` | **Date**: 2026-05-22 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/009-hooks-breakdown-tab/spec.md`

## Summary

Add a Hooks selector to the Now tab analytics breakdown that reports per-row
hook fire counts, mirroring the existing Skills selector. Claude rows come
from extracting `type:"attachment"` records with `attachment.type` beginning
`hook_` during the existing session indexing extraction pass — the dual
emission already happens for skills and session events, so this is a third
sibling extractor in the same parse. Codex rows come from a new tiny
`hook-observe.cjs` script deployed by the installer onto every Codex hook
event slot; the script POSTs to a new `/api/v1/hooks/observed` endpoint.
Both providers land in one new `hook_invocations` table behind a migration
27 + `hook_invocation_reingest_pending` flag, so existing Claude transcripts
backfill on next boot. The UI extends `BreakdownPanel.tsx` and
`useBreakdownData.ts` with a new mode, adds a `QUILL` chip for
Quill-deployed scripts, and reuses the existing All/Codex/Claude filter
strip and ALL TIME chip.

## Technical Context

**Language/Version**: Rust 2024 edition (backend), TypeScript 5.x (frontend),
Node.js (managed Codex hook scripts)
**Primary Dependencies**: Tauri 2, `rusqlite` (WAL mode, 5-second busy
timeout), `chrono`, Axum HTTP server on port 19876, React 19. No new
runtime dependencies.
**Storage**: SQLite at `~/.local/share/com.quilltoolkit.app/usage.db` —
adds one table at migration 27 (`hook_invocations`) and one boolean settings
key (`hook_invocation_reingest_pending`).
**Testing**: `cargo test` for the Rust storage + sessions + server changes
(modules carry `#[cfg(test)]` blocks); manual UI verification via
`npm run tauri dev`; quickstart walkthrough exercises the live ingest path.
**Target Platform**: Linux, macOS, Windows desktop (Tauri 2).
**Project Type**: desktop-app (Tauri 2: Rust backend + TS/React frontend +
deployed Node scripts that run inside provider CLI processes).
**Performance Goals**: Hook ingestion must not regress transcript indexing
throughput — the new extractor is a constant-time inspection of each
attachment record. The new `/api/v1/hooks/observed` endpoint must clear
acknowledgement inside the existing 1.5-second local-hook timeout used by
`observe.cjs` and `report-tokens.sh`. Hooks breakdown query for the 1h
window returns in under 100 ms on a typical machine.
**Constraints**: Migration must be safely re-runnable. The new table must
share per-session deletion semantics with `skill_usages` and `session_events`
so session-deletion stays atomic. No new background tasks or threads on the
backend. Codex script deployment must respect the existing
`activity_tracking` privacy gate. No changes to the Tantivy search index
schema. No changes to existing IPC commands; only a new `get_hook_breakdown`
command is added.
**Scale/Scope**: Claude transcripts on heavy developer machines hold tens of
thousands of hook attachment records per month (every tool use produces at
least one). Steady-state `hook_invocations` size is expected to land in the
low-to-mid millions of rows on a heavy machine within 90 days — within
SQLite's comfortable range with the proposed indices. Codex observed
events run at roughly one per turn plus one per tool call.
